 Flight Telemetry Analytics Pipeline
A production-style aerospace data pipeline that ingests raw UAV/STOL aircraft telemetry logs, cleans and indexes them at scale, detects anomalies using statistical methods, and uses an LLM (Claude API) to automatically generate human-readable fault summaries, root-cause hypotheses, and maintenance recommendations.
Built to mirror the real-world data challenges in autonomous aircraft systems — messy multimodal sensor streams, time-series anomaly detection, and LLM-powered log intelligence.
________________________________________
🎯 What Problem Does This Solve?
After every flight, an aircraft's onboard sensors produce thousands of rows of raw telemetry data — altitude, airspeed, motor temperatures, battery voltage, vibration, and attitude angles. Manually reviewing this data is slow, error-prone, and doesn't scale across a fleet.
This pipeline automates the full journey:
Raw CSV Logs (10 Hz sensor data)
        │
        ▼
 Stage 1 │ Ingest & Validate      →  Schema checks, null reporting, fail-fast errors
        │
        ▼
 Stage 2 │ Clean & Resample       →  Uniform time grid (integer-indexed to avoid float drift)
        │
        ▼
 Stage 3 │ Feature Engineering    →  Phase labels, rate-of-change, motor spread, rolling z-score
        │
        ▼
 Stage 4 │ Anomaly Detection      →  Threshold rules + statistical z-score (catches slow drifts)
        │
        ▼
 Stage 5 │ Metrics & Reporting    →  Per-flight KPIs, fleet-level JSON report, diagnostic plots
        │
        ▼
 LLM Layer │ Claude API           →  Fault signatures, root causes, maintenance recommendations

⚡ Quickstart
1. Clone & Install
git clone https://github.com/YOUR_USERNAME/flight-telemetry-pipeline.git
cd flight-telemetry-pipeline

pip install numpy pandas matplotlib scipy anthropic
2. Generate Simulated Flight Logs
python generate_logs.py
This creates 5 flights in data/ — FLT_003 and FLT_005 have a Motor 2 overheat fault injected between t=200s and t=350s (peaks at ~116°C), mimicking a bearing seizure or winding short.
3. Run the Full Pipeline
python pipeline.py
Outputs:
•	outputs/*_processed.csv — cleaned, feature-engineered telemetry for each flight
•	outputs/*_dashboard.png — 5-panel diagnostic plots with fault markers
•	outputs/flight_report.json — structured fleet-level KPI summary
4. Run LLM Analysis (requires Anthropic API key)
export ANTHROPIC_API_KEY=your_key_here

# Analyse all flights
python llm_summarizer.py

# Analyse a single flight
python llm_summarizer.py --flight FLT_003

📡 Telemetry Schema
Each flight log is a CSV with 10 Hz sensor data (10 readings per second):
Column	Type	Description
timestamp_s	float	Elapsed time in seconds since takeoff
flight_id	string	Unique flight identifier (e.g. FLT_003)
altitude_m	float	GPS altitude in metres
airspeed_ms	float	Indicated airspeed in m/s
pitch_deg	float	Pitch angle in degrees
roll_deg	float	Roll angle in degrees
yaw_deg	float	Yaw/heading in degrees
motor1_temp_c – motor4_temp_c	float	Per-motor temperature in °C
battery_v	float	Battery pack voltage (V)
battery_pct	float	State of charge (%)
vibration_g	float	Airframe vibration in g

🔍 Pipeline Deep Dive
Stage 2 — Why Integer Resampling?
A subtle but critical engineering decision: real sensor clocks drift slightly, so timestamps like 0.1, 0.2, 0.3 ... accumulate floating-point error over a long flight. Using np.arange() with float steps means the uniform reindex grid and the actual data timestamps never perfectly match — rows silently get dropped and replaced with interpolated values, corrupting fault data.
Fix: Convert timestamps to integer sample indices (timestamp × freq_hz), resample in integer space (no drift), then convert back to floats. This is a real production-grade bug — silent data corruption is far worse than a visible crash.
Stage 3 — Feature Engineering Highlights
Feature	Why it matters
altitude_ms_rate	Tells you what the aircraft is doing, not just where it is
motor_max_temp	Worst-case motor health across the whole system
motor_temp_spread	One motor at 110°C while others are at 60°C is more diagnostic than absolute temp alone
motor_temp_zscore	Rolling z-score over 50 samples — catches slow thermal drift before it hits a hard threshold
phase	Labels each row as ground / takeoff / climb / cruise / descent / landing
Stage 4 — Dual-Layer Anomaly Detection
Threshold rules (hard engineering limits):
Rule	Threshold	Flag
Motor temp	> 80°C	MOTOR_TEMP_WARN
Motor temp	> 95°C	MOTOR_TEMP_CRITICAL
Motor temp spread	> 25°C	MOTOR_TEMP_SPREAD_HIGH
Battery voltage	< 38V	LOW_BATTERY_VOLTAGE
Vibration	> 0.6g	HIGH_VIBRATION
Pitch / Roll	> 30° / 45°	EXCESSIVE_ATTITUDE
Statistical z-score (catches what rules miss):
•	Rolling window of 50 samples (~5 seconds)
•	Flag if |z| > 3.5σ
•	A motor slowly drifting from 55°C → 79°C over 10 minutes never crosses the 80°C threshold — but its z-score spikes early, giving you advance warning

🤖 LLM Integration
llm_summarizer.py sends each flight's structured JSON metrics to Claude with a system prompt that constrains output to a strict JSON schema — no free-form prose, no hallucinated numbers. The LLM does interpretation, not computation.
Example output for FLT_003:
{
  "flight_id": "FLT_003",
  "status": "CRITICAL",
  "summary": "Motor 2 experienced a progressive thermal excursion during mid-cruise, peaking at 116°C — 21°C above critical threshold — over a sustained 150-second window.",
  "fault_signatures": [
    "Motor 2 progressive overheat: 55°C → 116°C over 150s",
    "Motor temp spread reached 47°C — strong indicator of single-motor fault",
    "896 samples flagged across CRITICAL and SPREAD rules simultaneously"
  ],
  "root_cause_hypotheses": [
    "Bearing seizure on motor 2 spindle causing excess friction heat",
    "Phase winding short increasing resistive losses",
    "ESC misconfiguration causing motor 2 to draw excess current"
  ],
  "recommendations": [
    "Ground motor 2 immediately pending physical inspection",
    "Measure motor 2 phase-to-phase resistance vs motors 1, 3, 4",
    "Pull ESC telemetry logs for motor 2 current draw during the fault window"
  ],
  "performance_score": 34
}
The key design principle: the pipeline does the math, the LLM does the language. This makes the system reliable and auditable — every LLM statement is grounded in verified metrics.
